## 为什么有这个项目

用 AI 独立开发游戏非常强大——但一个聊天会话缺乏结构。没有人阻止你硬编码魔法数字、跳过设计文档、或者写出面条代码。没有 QA 审查、没有设计评审、没有人问"这真的符合游戏愿景吗？"

**Claude Code Game Studios** 通过给你的 AI 会话赋予真实工作室的结构来解决这个问题。你得到的不是一个通用助手，而是 48 个按工作室层级组织的专业代理——总监守护愿景，部门负责人掌控各自领域，专家负责具体执行。每个代理都有明确的职责、上报路径和质量关卡。

结果是：你仍然做出每一个决定，但现在你有一个团队会提出正确的问题、尽早发现错误，并让你的项目从最初的头脑风暴到发布都保持有序。

### 安装

1. **克隆或用作模板**：
   ```bash
   git clone https://github.com/Donchitos/Claude-Code-Game-Studios.git my-game
   cd my-game
   ```

2. **打开 Claude Code** 并启动会话：
   ```bash
   claude

   如果你已经知道自己需要什么，也可以直接跳转到特定技能：
   - `/brainstorm` — 从零开始探索游戏创意
   - `/setup-engine godot 4.6` — 如果你已经确定引擎，配置你的引擎
   - `/project-stage-detect` — 分析已有项目

## 工作原理

### 代理协调

代理遵循结构化的委派模型：

1. **垂直委派** ——总监委派给部门负责人，部门负责人委派给专家
2. **横向协商** ——同层级代理可以相互协商，但不能做出跨领域的约束性决策
3. **冲突解决** ——分歧上报至共同的上级（设计冲突上报至 `creative-director`，技术冲突上报至 `technical-director`）
4. **变更传播** ——跨部门的变更由 `producer` 协调
5. **领域边界** ——代理不修改其领域之外的文件，除非有明确委派

### 协作，而非自主

这**不是**一个自动驾驶系统。每个代理遵循严格的协作协议：

1. **提问** ——代理在提出解决方案之前先提问
2. **展示选项** ——代理展示 2-4 个方案及其利弊
3. **你做决定** ——用户始终做出最终决定
4. **草稿** ——代理在最终确认前展示工作成果
5. **审批** ——没有你的确认，不会写入任何内容

你始终掌控全局。代理提供结构和专业知识，而不是自主行动。

### 自动化安全

**钩子** 在每次会话中自动运行：

| 钩子 | 触发条件 | 功能 |
|------|----------|------|
| `validate-commit.sh` | `git commit` | 检查硬编码值、TODO 格式、JSON 有效性、设计文档章节 |
| `validate-push.sh` | `git push` | 向受保护分支推送时发出警告 |
| `validate-assets.sh` | 在 `assets/` 中写入文件 | 验证命名规范和 JSON 结构 |
| `session-start.sh` | 会话打开 | 加载冲刺上下文和最近的 git 活动 |
| `detect-gaps.sh` | 会话打开 | 检测新项目（建议运行 `/start`）以及代码/原型存在但缺少文档的情况 |
| `pre-compact.sh` | 上下文压缩 | 保留会话进度记录 |
| `session-stop.sh` | 会话关闭 | 记录工作成果 |
| `log-agent.sh` | 代理生成 | 所有子代理调用的审计日志 |

`settings.json` 中的**权限规则**自动允许安全操作（git status、测试运行），阻止危险操作（强制推送、`rm -rf`、读取 `.env` 文件）。

### 路径范围规则

编码标准根据文件位置自动强制执行：

| 路径 | 强制执行内容 |
|------|-------------|
| `src/gameplay/**` | 数据驱动值、delta time 使用、禁止 UI 引用 |
| `src/core/**` | 热路径零分配、线程安全、API 稳定性 |
| `src/ai/**` | 性能预算、可调试性、数据驱动参数 |
| `src/networking/**` | 服务器权威、版本化消息、安全性 |
| `src/ui/**` | 不持有游戏状态、支持本地化、无障碍访问 |
| `design/gdd/**` | 必须包含 8 个章节、公式格式、边界情况 |
| `tests/**` | 测试命名、覆盖率要求、测试夹具模式 |
| `prototypes/**` | 放宽标准、需要 README、记录假设 |

## 设计理念

本模板基于专业游戏开发实践：

- **MDA 框架** ——机制、动态、美学分析用于游戏设计
- **自我决定理论** ——自主性、能力感、关联性用于玩家动机
- **心流状态设计** ——挑战与技能的平衡用于玩家参与度
- **巴特尔玩家类型** ——目标受众定位与验证
- **验证驱动开发** ——先写测试，再实现功能

## 自定义

这是一个**模板**，不是锁定的框架。一切都可以自定义：

- **调整钩子** ——调整验证严格程度，添加新检查
- **选择引擎** ——使用 Godot、Unity 或 Unreal 代理集（或都不用）

## 阶段 0：设置与配置

### 你需要什么

开始之前，请确保你拥有：

- **Claude Code** 已安装并可正常使用
- **Git**，搭配 Git Bash（Windows）或标准终端（Mac/Linux）
- **jq**（可选但推荐 — 缺少时钩子会回退到 `grep`）
- **Python 3**（可选 — 部分钩子使用它进行 JSON 验证）

### 步骤 0.1：克隆并配置

克隆仓库并在编辑器中打开：

```bash
git clone <repo-url> my-game
cd my-game
```

### 步骤 0.2：运行 /start（推荐新用户使用）

如果你是项目新手或还不知道要做什么游戏：

```
/start
```

这个引导式入门流程会询问你目前的状态（没有想法、模糊想法、清晰概念、
已有作品）并将你引导到正确的阶段。如果你已经有游戏概念和引擎选择，可以跳过。

### 步骤 0.3：选择你的引擎

在 Claude Code 中运行 `/setup-engine`。这是最重要的单一
配置步骤 — 它告诉每个代理你使用的引擎、语言和工具链：

```bash
/setup-engine godot 4.6
```

或者不带参数运行 `/setup-engine`，根据你的游戏需求（2D/3D、平台、团队规模、
语言偏好）获得交互式推荐。

**`/setup-engine` 的作用：**

- 在 `CLAUDE.md` 中锁定引擎和版本
- 用命名约定、性能预算和引擎特定默认值填充 `.claude/docs/technical-preferences.md`
- 检测知识缺口（引擎版本比 LLM 训练数据更新）并从网络获取最新文档，
  以确保代理推荐正确的 API
- 在 `docs/engine-reference/` 中创建版本锁定的参考文档

**为什么这很重要：** 一旦设定了引擎，系统就知道该使用哪些引擎专家代理。
如果你选择了 Godot，`godot-specialist`、`godot-gdscript-specialist` 和
`godot-shader-specialist` 等代理将成为你的首选专家。Unity 和 Unreal 专家
仍然可用，但不会是主要选择。

> **手动替代方案：** 如果你愿意，也可以直接编辑 `CLAUDE.md` 中的技术栈部分。

### 步骤 0.3：验证钩子是否工作

启动一个新的 Claude Code 会话。你应该看到 `session-start.sh` 钩子的输出：

```
=== Claude Code Game Studios -- 会话上下文 ===
分支：main
最近提交：
  abc1234 初始提交
===================================
```

如果你看到了这个，说明钩子工作正常。如果没有，请检查 `.claude/settings.json`
确保钩子路径对你的操作系统是正确的。

### 步骤 0.4：创建你的目录结构

## 阶段 2：前期制作与设计

### 本阶段内容

你创建定义游戏运作方式的所有设计文档。此时还不写代码 —
纯粹是设计和架构。

### 步骤 2.1：将概念分解为系统

在编写单个 GDD 之前，枚举游戏需要的所有系统：

```
/map-systems
```

这会创建 `design/gdd/systems-index.md` — 一个主跟踪文档，它：
- 列出游戏需要的每个系统（战斗、移动、UI 等）
- 映射系统之间的依赖关系
- 分配优先级层级（MVP、垂直切片、Alpha、完整愿景）
- 确定设计顺序（基础 → 核心 → 功能 → 表现 → 打磨）

然后按依赖顺序设计每个系统：

```
/map-systems next
```

这会选取最高优先级的未设计系统，交给 `/design-system`，
引导你逐章节创建其 GDD。每个完成的 GDD 在下一个开始前都要经过 `/design-review`。

你也可以直接编写特定系统的 GDD：

```
/design-system combat-system
```

### 步骤 2.2：创建游戏设计文档（GDD）

为游戏中的每个主要系统，在 `design/gdd/` 中创建设计文档。
使用模板：

```
让 Claude 使用 .claude/docs/templates/game-design-document.md 模板
创建 GDD
```

**`design-docs` 规则**（`.claude/rules/design-docs.md`）要求每个 GDD
具有以下 8 个部分：

| # | 章节 | 内容 |
|---|------|------|
| 1 | **概述** | 系统的一段式摘要 |
| 2 | **玩家幻想** | 玩家使用此系统时的想象/感受 |
| 3 | **详细规则** | 无歧义的机械规则 — 如果两个设计师读这个，他们会构建出同样的东西 |
| 4 | **公式** | 每个计算，带有变量定义和范围 |
| 5 | **边缘情况** | 奇怪情况下会发生什么？明确解决。 |
| 6 | **依赖项** | 连接到哪些其他系统（双向） |
| 7 | **调节旋钮** | 设计师可以安全更改的值，附带安全范围 |
| 8 | **验收标准** | 如何测试此功能是否正常？具体、可衡量。 |

**你可能创建的示例 GDD 文件：**
- `design/gdd/core-loop.md` — 基本游戏循环
- `design/gdd/combat-system.md` — 战斗系统运作方式
- `design/gdd/progression-system.md` — 玩家如何进阶
- `design/gdd/inventory-system.md` — 物品管理
- `design/gdd/ability-system.md` — 玩家能力和技能

### 步骤 2.2：使用游戏设计师代理处理复杂机制

对于需要深度机械思考的系统，委托给 `game-designer`：

```
让 game-designer 代理设计一个制作系统，支持涌现配方
（玩家发现组合），技能进度与实验挂钩。系统应该有回报感但不会枯燥。
目标受众是探索者和成就者。
```

**过程：** `game-designer` 代理与你协作创建设计：

代理："我来帮你设计制作系统。首先，几个问题：
       - 配方应该通过实验发现还是明确教导？
       - 失败的制作尝试惩罚应该多重？
       - 目标受众 — 主要是探索者还是成就者？"

用户："通过实验发现。中等惩罚 — 浪费材料但获得部分技能经验。主要面向探索者。"

       **方案 A：纯发现** — 无提示，完全自由

       **方案 B：提示系统** — 材料兼容时显示视觉提示

**最后更新：** 2026-02-13

## 🎯 核心理念

本代理(Agent)架构旨在实现**用户驱动的协作**，而非自主的 AI 生成。

### ✅ 正确模式：协作顾问

```
代理 = 专家顾问
用户 = 创意总监（最终决策者）

代理：
- 提出澄清问题
- 调研并呈现选项
- 解释权衡和推理过程
- 起草提案供审阅
- 在写入前等待用户批准

用户：
- 做出所有创意和战略决策
- 批准或否决代理建议
- 指导设计愿景
- 在内容写入文件前签字确认
```

### ❌ 错误模式：自主生成器

```
❌ 代理自行创建设计并写入文件
❌ 代理在没有用户输入的情况下做出决策
❌ 代理在未经批准的情况下编写代码
❌ 用户只需说"做个游戏"然后等待
```

---

## 🔄 协作工作流模式

每个代理交互都应遵循以下模式：

### 模式：提问 → 选项 → 决策 → 草稿 → 批准

```
1. 代理提出问题
   代理："为了设计制作系统，我需要了解：
          - 配方应该是通过发现获得还是从 NPC 处学习？
          - 失败的尝试应该有多大的惩罚？
          - 实验是否需要消耗资源？"

2. 用户提供上下文
   用户："配方通过实验发现。失败的尝试消耗材料但给予部分经验。
         我们主要面向探索者(Explorer)。"

3. 代理呈现带推理的选项
   代理："根据您的目标，这里有三种方案：

          **方案 A：完全随机发现**
          - 机制：尝试任意物品组合，随机成功概率
          - ✅ 最大化探索奖励
          - ❌ 缺乏技能表达时可能感觉随意
          - 参考游戏：《我的世界》早期、《上帝也疯狂》
          - 与支柱对齐：✅ 涌现故事，⚠️ 有意义的选择（低）

          **方案 B：成分提示系统**
          - 机制：物品检查时显示"兼容标签"
          - ✅ 奖励观察和推理
          - ✅ 通过模式识别实现技能表达
          - ❌ 纯粹的发现感较弱
          - 参考游戏：《药剂工艺》(Potion Craft)、《无限运算机》(Opus Magnum)
          - 与支柱对齐：✅ 有意义的选择，✅ 涌现故事

          **方案 C：混合模式（发现 + 提示）**
          - 机制：开始时随机，X 次失败后解锁提示系统
          - ✅ 适合多种玩家类型
          - ✅ 奖励坚持
          - ❌ 实现更复杂
          - 与支柱对齐：✅✅ 两个支柱都强烈对齐

          哪种方案最符合您的愿景？"

4. 用户决策
   用户："方案 C 感觉对了。设 X = 5 次失败后解锁提示。"

5. 代理基于决策起草
   代理："好的。以下是核心机制的草稿：

          ### 核心循环
          1. 玩家从背包中选择 2-4 个材料
          2. 系统与配方数据库进行比对
          3. 匹配成功：配方习得，物品创建，获得完整经验
          4. 无匹配：失败。材料消耗，获得 25% 经验
          5. 对同一物品组合失败 5 次后，为这些物品解锁提示系统
          6. 提示显示材料的"标签"（如"火"、"治疗"、"金属"）

          ### 公式提案
          success_xp = base_xp * recipe_tier^1.5
          failure_xp = success_xp * 0.25
          hint_unlock_threshold = 5  # 解锁提示所需的失败次数

          这符合您的意图吗？"

6. 用户审阅并完善
   用户："循环看起来不错，但 25% 的经验太慷慨了。改为 15%。
         另外，让提示阈值有梯度：1 级配方 5 次，2 级及以上 3 次，
         这样玩家不会因为复杂的配方而沮丧。"

7. 代理更新草稿
   代理："已更新：
          failure_xp = success_xp * 0.15
          hint_unlock_threshold = 5 if recipe_tier == 1 else 3

          我是否继续起草边缘情况(E Edge Cases)部分，还是您想先更详细地审查公式？"

8. 用户批准下一步
   用户："公式没问题。继续处理边缘情况。"

9. [对每个部分重复此循环]

10. 写入前的最终批准
    代理："完整的 GDD 草稿已完成。我已涵盖：
           ✓ 概述、✓ 玩家幻想、✓ 详细规则、✓ 公式、
           ✓ 边缘情况、✓ 依赖项、✓ 调节旋钮、✓ 验收标准

           我可以将此写入 design/gdd/crafting-system.md 吗？"

    用户："可以，写吧。"  ← 只有现在才创建文件

11. 代理写入文件
    代理：[使用 Write 工具]
           "已创建 design/gdd/crafting-system.md。您是否希望我运行
            /design-review 来验证其是否符合标准？"
```

## 🛠️ 代理应该如何提问

### 好的提问模式

✅ **带推理的多选题：**
```
"敌人是否应该有攻击预警？
 A) 是的，0.5 秒前（可及性强，节奏导向）
 B) 是的，0.2 秒前（时机紧凑，技能导向）
 C) 没有预警（纯模式学习，高难度）
 哪种符合你的愿景？"
```

✅ **带权衡的约束选项：**
```
"背包系统选项：
 1. 网格式（《生化危机》、《暗黑破坏神》）：空间管理深度高，速度较慢
 2. 列表式（《上古卷轴》、《辐射》）：快速访问，策略性较低
 3. 混合式（重量限制 + 有限槽位）：中等复杂度

 鉴于你的'有意义的选择'支柱，我倾向于方案 1 或 3。你怎么看？"
```

✅ **带上下文的开放式问题：**
```
"设计文档没有指定玩家在制作过程中死亡会发生什么。
 一些选项：
 - 材料丢失（严厉，风险/回报）
 - 材料退回背包（宽松）
 - 进行中的工作保存（实现复杂）

 哪种符合你的目标难度？"
```

### 不好的提问模式

❌ **过于开放：**
```
"战斗系统应该是什么样的？"
← 太宽泛，用户不知道从何开始
```

❌ **引导/预设：**
```
"我将把战斗做成实时的，因为这是这个类型的标准做法。"
← 没有提问，直接假设了
```

❌ **无上下文的二选一：**
```
"我们要不要做技能树？是或否？"
← 没有优缺点，没有参考游戏支柱
```

## 🎛️ 结构化决策 UI（AskUserQuestion）

使用 `AskUserQuestion` 工具将决策呈现为**可选择的 UI**，而不是纯 Markdown 文本。
这为用户提供了一个简洁的界面来选择选项（或输入"其他"来自定义回答）。

### 解释 → 捕获模式

详细的推理不适合放在工具的简短描述中。因此使用两步模式：

1. **先解释** — 在对话文本中写出完整的专家分析：
   详细的优缺点、理论参考、参考游戏、支柱对齐。这是推理所在的地方。

2. **捕获决策** — 调用 `AskUserQuestion`，使用简洁的选项标签
   和简短的描述。用户从 UI 中选择或输入自定义答案。

### 何时使用 AskUserQuestion

✅ **适用于：**
- 每个需要展示 2-4 个选项的决策点
- 答案受约束的初始澄清问题
- 将最多 4 个独立问题批量处理在一次调用中

# 示例会话：设计制作系统

**上下文：** 独立游戏开发者单人项目，项目第 2 周，需要设计游戏支柱中提到的制作系统。

**起始状态：**
- 游戏概念和支柱已定义
- 引擎已搭建（Godot 4.6）
- 核心战斗系统已设计但尚未实现
- 支柱 2："通过实验实现涌现式发现"

**会话时长：** ~45 分钟
**轮次：** 12
**成果：** 完整的制作系统 GDD（游戏设计文档），可进入实现阶段

---

## 第 1 轮：用户发起设计

**用户：**
> 我需要设计制作系统。支柱 2 是"通过实验实现涌现式发现"，所以我希望玩家通过尝试组合来发现配方。

## 第 2 轮：代理提出澄清问题

**Agent (game-designer)：**
> 我来帮你设计以发现为核心的制作系统。在提出机制之前，我需要了解你的愿景：
>
> 1. **发现方式**：配方应该是纯试错，还是应该有提示/线索引导玩家？
> 2. **失败惩罚**：当玩家尝试无效组合时，会发生什么？
>    - 材料完全损失（严厉）
>    - 材料部分退还（中等）
>    - 实验无成本（宽容）
> 3. **成长体系**：制作技能会随时间提升吗？如果是，更高技能解锁什么？
> 4. **范围**：制作在游戏中有多核心？是核心支柱功能还是辅助系统？
> 5. **参考游戏**：有没有你特别喜欢或讨厌的制作系统？（Minecraft、Terraria、Stardew Valley、Potion Craft？）

## 第 3 轮：用户提供上下文

**用户：**
> 1. 我想要逐步解锁的提示——开始时纯发现，通过技能获取提示
> 2. 中等惩罚——浪费材料但获得部分经验值（XP）
> 3. 是的，技能解锁提示系统并提高成功率
> 4. 核心功能——三大主要循环之一（战斗、探索、制作）
> 5. 喜欢 Potion Craft 的推理机制。讨厌 Minecraft 必须查 Wiki。

## 第 4 轮：代理展示选项

**Agent：**
> 上下文非常清晰！基于"涌现式发现" + "推理而非维基" + "Potion Craft 灵感"，以下是三种机制方案：
>
> **方案 A：纯随机发现**
> - **机制**：尝试任意 2-4 种材料组合，根据兼容性随机判定成功
> - **提示系统**：无——玩家手动记录发现
> - **优势**：
>   - 最大的涌现式发现感
>   - 社区可以分享发现而不破坏核心循环
>   - 与"实验"支柱高度契合
> - **劣势**：
>   - 完美主义者容易受挫
>   - 技能表达空间低（主要是随机性）
>   - 玩家可能觉得"必须查 Wiki"，尽管你并非此意
> - **支柱对齐**：✅✅ 涌现式发现，⚠️ 有意义的选择（随机性降低自主性）
>
> **方案 B：基于标签的推理系统（Potion Craft 风格）**
> - **机制**：每种材料有隐藏的"标签"（火、水、治愈、伤害等）
>   - 配方需要特定的标签组合
>   - 技能 0：标签不可见，纯实验
>   - 技能 5：可以查看物品的 1 个标签
>   - 技能 10：查看 2 个标签，以此类推
> - **提示系统**：标签本身就是提示系统，通过技能解锁
> - **优势**：
>   - 奖励观察力和模式识别（技能表达！）
>   - 渐进式脚手架——对所有玩家类型友好
>   - 当模式被识别时创造"顿悟！"时刻
>   - 无需 Wiki——所有内容皆可在游戏中发现
> - **劣势**：
>   - 实现更复杂（标签系统 + UI）
>   - 需要精心设计标签以避免退化策略
> - **支柱对齐**：✅✅ 涌现式发现，✅✅ 有意义的选择
>
> **方案 C：材料兼容性矩阵**
> - **机制**：每种材料与其他材料有良好/不良的配对关系
>   - 成功率 = 兼容性分数
>   - 技能 0：不显示兼容性，基础成功率约 30%
>   - 更高技能：显示颜色编码的兼容性（红/黄/绿）
> - **提示系统**：视觉兼容性指示器随技能解锁
> - **优势**：
>   - 实现和理解简单
>   - 清晰的成长弧线（技能 = 更多信息）
>   - 感觉公平（玩家理解失败原因）
> - **劣势**：
>   - 涌现性不如标签系统（更具规定性）
>   - 获得提示后最优配方更明显
> - **支柱对齐**：✅ 涌现式发现，✅ 有意义的选择
>
> **我的推荐：方案 B（基于标签的推理）**
>
> **推理：**
> - 这是唯一能通过系统交互创造真正涌现式发现的方案
> - 玩家通过学习标签模式来发展精通度，而非死记配方
> - 渐进式揭示（标签随技能解锁）同时服务于探索型玩家（发现）和成就型玩家（成长）
> - Potion Craft 已证明该机制有效且令人满足
> - 避免 Wiki 依赖，因为推理过程完全可以在游戏中完成
>
> 哪个方案符合你的愿景？还是我应该探索混合方案？

# 逆向文档工作流示例

## 场景

开发者构建了一个可用的技能树系统，但从未编写设计文档。
代码存在于 `src/gameplay/skills/`，但 `design/gdd/` 中没有 `skill-system.md`。

---

## 会话记录

**用户：**
```
我在 src/gameplay/skills/ 中有一个可用的技能树系统，大约 1200 行代码。
我从未编写过设计文档。能否从实现中创建一份？
```

**Game-Designer（读取代码，分析）：**
```
我已分析了你的技能树实现。以下是我的发现：

结构：
- 3 棵技能树（Skill Tree）：战斗、生存、科技
- 每棵树 5 个层级（Tier）（通过等级 + 前置条件解锁）
- 共 45 个技能（每棵树 15 个）
- 消耗：技能点（每升 1 级获得 1 点，可花费金币重置专精）

机制：
- 被动技能（Passive Skill）（属性加成）
- 主动技能（Active Skill）（带冷却时间的技能）
- 协同加成（Synergy Bonus）（4 层以上的跨树解锁）

实现细节：
- 数据驱动（JSON 文件位于 `assets/data/skills/`）
- 存档兼容（技能状态已序列化）
- UI 显示已锁定/已解锁/可解锁状态

在起草设计文档之前，有几个问题：

1. **设计意图**：3 棵树的结构是否意在代表玩家原型
   （战士/生存者/工程师）还是玩法风格的灵活性？

2. **重置消耗**：重置需要花费金币——这是否意在：
   - 阻止频繁重置（鼓励对构建方案的承诺）？
   - 创造金币回收机制（经济平衡）？
   - 还是应该免费（鼓励实验）？

3. **协同系统**：跨树协同在 4 层出现。这是：
   - 刻意为之（奖励后期的混合构建）？
   - 还是自然涌现（意外创造了混合流派 Meta）？

4. **平衡理念**：我看到某些技能明显更强（5 层被动是 +50% 伤害）。这是：
   - 力量幻想（后期应该感觉超模 Overpowered）？
   - 还是未平衡（需要调优）？
```

**用户：**
```
好问题！以下是我的意图：

1. 是的，3 棵树代表原型，但我希望保持灵活性
2. 重置应该便宜——我希望鼓励实验
3. 协同是刻意设计的——混合构建是一个特性
4. 是的，5 层应该很强大，但 +50% 可能太多了
```

**Game-Designer：**
```
很好，这澄清了设计愿景。让我起草设计文档：

[展示 `design/gdd/skill-system.md` 的草稿]

章节：
1. 概述（3 树系统、混合友好、鼓励重置）
2. 设计支柱（Design Pillar）：
   - 原型清晰度（每棵树有独特身份）
   - 构建灵活性（无锁定，重置便宜）
   - 协同奖励（跨树组合可行）
3. 详细设计（树结构、解锁规则、重置）
4. 平衡框架（层级威力曲线、协同限制）
5. 边界情况（战斗中重置？最大技能数？上限？）
6. 依赖关系（UI、存档系统、教程）
7. 验收标准（所有树可行、10+ 种混合构建、清晰的提示文本 Tooltip）
8. 待解决问题：
   - 重置消耗应随等级递增还是保持固定？
   - 5 层伤害加成：调至 +30% 还是保持 +50%？

**可以将此内容写入 `design/gdd/skill-system.md` 吗？**
```

**用户：**
```
可以，写入吧。但把 5 层调至 +30%，并让重置消耗随等级递增。
```

**Game-Designer：**
```
✅ 已写入 `design/gdd/skill-system.md`
✅ 标记为 [从实现逆向文档化 REVERSE-DOCUMENTED FROM IMPLEMENTATION]
✅ 添加待办：在 `assets/data/skills/combat-tree.json` 中将 5 层被动更新为 +30%
✅ 添加待办：实现递增重置消耗公式

后续步骤：
- 运行 /balance-check 验证技能树成长曲线？
- 创建架构决策记录（ADR），记录为什么选择 3 棵树而非 5 棵？
- 更新教程以解释混合协同？
```

## 关键观察

1. **代理提出了澄清性问题**——不只是描述代码
2. **区分意图与实现**——用户纠正了"意外"的不平衡
3. **设计文档捕捉了"为什么"**——不只是"代码做了什么"
4. **识别了缺失部分**——代码中未处理的边界情况
5. **标记了后续工作**——平衡调优、架构决策记录、教程

## 结果

设计文档现已存在，且**与现实一致**，同时捕捉了**设计愿景**。
未来的变更将引用此文档，新团队成员也能理解决策背后的"为什么"。

# 引擎参考文档

本目录包含为项目中使用的游戏引擎精选的、版本锁定的文档快照。
这些文件存在的原因是**LLM 知识有截止日期**，而游戏引擎频繁更新。

## 为什么需要这个

Claude 的训练数据有一个知识截止日期（当前为 2025 年 5 月）。
Godot、Unity 和 Unreal 等游戏引擎不断发布更新，引入破坏性 API 变更、
新功能和废弃的模式。如果没有这些参考文件，代理(Agent)会建议过时的代码。

## 结构

每个引擎有自己的目录：

```
<engine>/
├── VERSION.md              # 锁定版本、验证日期、知识缺口窗口
├── breaking-changes.md     # 版本间的 API 变更，按风险级别组织
├── deprecated-apis.md      # "不要用 X → 改用 Y" 查找表
├── current-best-practices.md  # 模型训练数据中没有的新实践
└── modules/                # 按子系统的快速参考（每个最多 ~150 行）
    ├── rendering.md
    ├── physics.md
    └── ...
```

## 代理如何使用这些文件

引擎专家代理被指示执行：

1. 读取 `VERSION.md` 确认当前引擎版本
2. 在建议任何引擎 API 之前检查 `deprecated-apis.md`
3. 查阅 `breaking-changes.md` 了解版本特定问题
4. 读取相关的 `modules/*.md` 进行子系统特定工作

## 维护

### 何时更新

- 升级引擎版本后
- LLM 模型更新后（新的知识截止日期）
- 运行 `/refresh-docs` 后（如可用）
- 发现模型弄错的 API 时

### 如何更新

1. 在 `VERSION.md` 中更新新的引擎版本和日期
2. 为版本过渡在 `breaking-changes.md` 中添加新条目
3. 将新废弃的 API 移入 `deprecated-apis.md`
4. 用新模式更新 `current-best-practices.md`
5. 用 API 变更更新相关的 `modules/*.md`
6. 在所有修改的文件上设置"最后验证"日期

### 质量规则

- 每个文件必须有"最后验证：YYYY-MM-DD"日期
- 保持模块文件在 150 行以内（上下文预算）
- 包含代码示例展示正确/错误模式
- 链接到官方文档 URL 以供验证
- 只记录与模型训练数据不同的内容

# Unity 6.3 LTS — 破坏性变更

**最后验证:** 2026-02-13

本文档追踪 Unity 2022 LTS（可能在模型训练数据中）与 Unity 6.3 LTS（当前版本）之间的破坏性 API 变更和行为差异。按风险等级组织。

## 高风险 — 将破坏现有代码

### Entities/DOTS API 完全重构
**版本:** Entities 1.0+ (Unity 6.0+)

```csharp
// ❌ 旧版（Unity 6 之前，GameObjectEntity 模式）
public class HealthComponent : ComponentData {
    public float Value;
}

// ✅ 新版（Unity 6+，IComponentData）
public struct HealthComponent : IComponentData {
    public float Value;
}

// ❌ 旧版: ComponentSystem
public class DamageSystem : ComponentSystem { }

// ✅ 新版: ISystem（非托管，Burst 兼容）
public partial struct DamageSystem : ISystem {
    public void OnCreate(ref SystemState state) { }
    public void OnUpdate(ref SystemState state) { }
}
```

**迁移:** 遵循 Unity 的 ECS 迁移指南。需要重大架构变更。

---

### 输入系统 — 旧版 Input 已弃用
**版本:** Unity 6.0+

```csharp
// ❌ 旧版: Input 类（已弃用）
if (Input.GetKeyDown(KeyCode.Space)) { }

// ✅ 新版: Input System 包
using UnityEngine.InputSystem;
if (Keyboard.current.spaceKey.wasPressedThisFrame) { }
```

**迁移:** 安装 Input System 包，将所有 `Input.*` 调用替换为新 API。

---

### URP/HDRP 渲染器特性 API 变更
**版本:** Unity 6.0+

```csharp
// ❌ 旧版: ScriptableRenderPass.Execute 签名
public override void Execute(ScriptableRenderContext context, ref RenderingData data)

// ✅ 新版: 使用 RenderGraph API
public override void RecordRenderGraph(RenderGraph renderGraph, ContextContainer frameData)
```

**迁移:** 更新自定义渲染通道以使用 RenderGraph API。

---

## 中等风险 — 行为变更

### Addressables — 资源加载返回值变更
**版本:** Unity 6.2+

资源加载失败时现在默认抛出异常，而不是返回 null。
需要添加适当的异常处理，或使用 `TryLoad` 变体。

```csharp
// ❌ 旧版: 失败时静默返回 null
var handle = Addressables.LoadAssetAsync<Sprite>("key");
var sprite = handle.Result; // 失败时为 null

// ✅ 新版: 失败时抛出异常，使用 try/catch 或 TryLoad
try {
    var handle = Addressables.LoadAssetAsync<Sprite>("key");
    var sprite = await handle.Task;
} catch (Exception e) {
    Debug.LogError($"加载失败: {e}");
}
```

---

### 物理 — 默认求解器迭代次数变更
**版本:** Unity 6.0+

默认求解器迭代次数已增加以提高稳定性。
如果你依赖旧行为，请检查 `Physics.defaultSolverIterations`。

---

## 低风险 — 已弃用（仍可使用）

### UGUI（旧版 UI）
**状态:** 已弃用但仍受支持
**替代方案:** UI Toolkit

UGUI 仍然可用，但推荐 UI Toolkit 用于新项目。

---

### 旧版粒子系统
**状态:** 已弃用
**替代方案:** Visual Effect Graph (VFX Graph)
